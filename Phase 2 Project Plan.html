Capstone Domain Guardian

Phase 2 Project Plan

Purpose

Phase 2 will move the current add-on from a working proof of concept into a more complete analyst tool. The focus is to close the main gaps in detection quality, infrastructure maturity, design clarity, and reporting output without changing the core manual-analysis workflow.

1. Current State

The current version already supports manual URL/domain analysis, brand override, deterministic scoring, analyst-facing evidence display, JSON export, DNSTwister-based lookalike generation, and an AWS-oriented backend structure.

The main issue is not that the application is non-functional. The issue is that several important services are still partial, heuristic, or operationally immature.

2. Main Gaps Identified

Technical

  • Some external intelligence providers are implemented but still depend on local configuration and can return degraded results.
  • Infrastructure signals still contain partial or null fields such as DNS history change count, IP reputation score, and redirect count.
  • Content analysis is still mostly heuristic and does not yet provide strong phishing-template or visual-similarity evidence.
  • Scoring weights are hand-tuned and not yet calibrated against a formal labeled sample set.
  • The test suite covers important units, but not the full analyst workflow end to end.

Infrastructure

  • The AWS stack is synth-ready, but there is no complete deployment and release workflow for dev, test, and prod.
  • Secrets handling, provider readiness, dashboards, and alerts need to be hardened for actual use.
  • CDK deprecations remain in the infrastructure code and should be cleaned up before a stable release.

Design

  • The UI displays the data, but not always with the best hierarchy for analyst triage.
  • The difference between domain risk and tooling degradation is still not as clear as it should be.
  • Lookalike-generation results are usable, but not yet optimized for batch analyst review.

Reporting

  • Reporting contacts are present, but coverage is still limited.
  • There is no structured analyst report output beyond raw JSON export.
  • There is no audience-specific reporting workflow for registrars, brands, CERT/CSIRT, or internal security teams.

3. Phase 2 Objectives

  1. Improve detection depth and evidence quality.
  2. Harden the backend and deployment model for stable use.
  3. Improve the extension workflow for analyst triage and review.
  4. Produce reporting outputs that are actionable outside the application.

4. Workstream Plan

4.1 Technical Workstream

Goal: Improve the quality, completeness, and reliability of the detection pipeline.

  • Harden VirusTotal, AbuseIPDB, Google Safe Browsing, PhishTank, and OpenPhish integrations.
  • Improve timeout, rate-limit, and upstream-failure handling for external providers.
  • Implement redirect-chain counting and final-destination classification.
  • Replace placeholder infrastructure values such as IP reputation score with real signal logic.
  • Expand passive history and ownership-history intelligence.
  • Strengthen content analysis with better visible-text similarity, asset reuse checks, and form-action classification.
  • Evaluate stronger phishing-template and visual-similarity detection.
  • Expand brand-catalog coverage and improve typo and confusable-character matching.
  • Calibrate scoring using a labeled set of benign and malicious samples.

4.2 Infrastructure Workstream

Goal: Make the backend operationally stable and deployable as a real service.

  • Formalize AWS environments for dev, test, and prod in ca-central-1.
  • Finalize provider-key management using Secrets Manager or SSM.
  • Clean up CDK deprecations in the current stack.
  • Validate DynamoDB cache lifecycle and retention behavior.
  • Expand CloudWatch alarms, dashboards, and log retention policies.
  • Define a repeatable release process for backend deployment and extension packaging.
  • Package the extension against the deployed API instead of a local-only address.
  • Add smoke-test procedures for post-deployment verification.

4.3 Design and Analyst Experience Workstream

Goal: Make the extension easier to use as a triage tool without changing the core functionality.

  • Separate high-confidence findings from degraded-provider warnings more clearly.
  • Improve prioritization and grouping of evidence.
  • Improve empty, error, timeout, and degraded-result states.
  • Tighten popup layout and repeated-use ergonomics.
  • Improve the DNSTwister candidate-review workflow for batch analysis.
  • Align status hierarchy, spacing, and typography across all analyst views.

4.4 Reporting Workstream

Goal: Turn the analysis output into something usable for external reporting and documentation.

  • Expand registrar abuse contacts and reporting portals.
  • Expand brand-contact coverage where public security contacts exist.
  • Improve authority routing for Canadian and general reporting use cases.
  • Reduce dependency on LLM-generated notes for core reporting details.
  • Create a human-readable analyst report format in addition to JSON export.
  • Support reporting views for registrars, brand owners, CERT/CSIRT contacts, and internal teams.

5. Milestone Breakdown

Milestone Focus Primary Output
Milestone 1 Provider hardening, infrastructure intelligence completion, passive-history expansion, provider diagnostics cleanup Reliable backend evidence model
Milestone 2 Content-analysis improvements, stronger phishing-template detection, brand-matching expansion, score calibration Higher-quality detection and defensible scoring
Milestone 3 Analyst UX cleanup, lookalike-review improvements, structured report output, expanded reporting coverage Usable analyst workflow and reporting package
Milestone 4 AWS environment maturity, dashboards, release packaging, expanded tests, deployment validation Stable Phase 2 release candidate

6. Glaringly Obvious Features Still Missing

  • True redirect-count and destination-risk analysis
  • A real IP reputation score in the infrastructure model
  • Deeper passive DNS and ownership-history intelligence
  • Stronger visual similarity or phishing-template detection
  • A human-readable exported analyst report
  • Broader registrar and brand reporting coverage
  • End-to-end UI automation for the extension
  • A fully documented production deployment and release workflow
  • Calibrated scoring based on a maintained sample corpus

7. Recommended Delivery Order

  1. Complete provider hardening and infrastructure intelligence.
  2. Upgrade content analysis and calibrate scoring.
  3. Improve reporting outputs and contact coverage.
  4. Polish the analyst UI and lookalike review workflow.
  5. Finalize deployment, release workflow, and expanded testing.

8. Definition of Done

  • All surfaced signals are backed by real logic or an explicit supported failure state.
  • Degraded provider conditions are clearly distinguishable from actual domain risk.
  • Infrastructure and content evidence are strong enough to support analyst conclusions.
  • The extension supports clear triage and reporting workflows.
  • Exported reporting is useful to someone outside the application.
  • The backend can be deployed and validated without relying on a local-only setup.
  • The project has test coverage for both normal and degraded operating conditions.
